## Summary

This phase is a documentation audit, not a code-writing task. The executor must cross-reference three sources: TECH-STORIES.md (13 stories, ~880 lines), REQUIREMENTS.md (PRD with v1 + v2 requirements), and the live MVP codebase (Next.js app with Supabase). The research below characterises exactly what is in each source and where alignment gaps already exist, so the planner can produce precise, actionable tasks.

The MVP was built as a prototype/demo application. It does not implement the queue-based architecture described in many Tech Stories (no COMPUTE_SCORE tasks, no RERANK_HOTELS tasks, no SEND_FEEDBACK_REQUEST tasks). Instead, scoring, ranking, and sentiment analysis all run inline at submission time or at config-save time. This is a pervasive and significant deviation between story text and MVP reality, but it does not violate PRD intent — the PRD was written against the delivered prototype behaviour.

The TECH-STORIES.md document describes a production-grade system design (task queues, scheduled jobs, separate workers, signed tokens, RBAC). The MVP implements a simplified version of that design. The planner must ensure the VERIFICATION-REPORT.md treats these as "Minor gaps" (PRD satisfied, implementation differs from story) rather than "Critical gaps," because the PRD requirements are all met.

**Primary recommendation:** The planner should create one task that reads all three sources and writes VERIFICATION-REPORT.md, plus a second task that applies auto-corrections directly to TECH-STORIES.md (cosmetic fixes and any new TS-14+ stories for critical gaps). These are sequentially dependent — report first, corrections second.

## Architecture Patterns

### Pattern 1: Inline Synchronous vs. Async Queue Architecture

The single most important architectural fact for the executor to understand:

**TECH-STORIES.md describes a queue-based async architecture.** Tasks like `COMPUTE_SCORE`, `RERANK_HOTELS`, `SEND_FEEDBACK_REQUEST`, and `ANALYZE_SENTIMENT` are enqueued and consumed by workers.

**The MVP implements synchronous inline operations.** All of these "tasks" happen synchronously within server actions or server components at request time.

This is a global "Minor gap" pattern affecting TS-1 through TS-13. Each story's gap entry in the report should reference this architectural divergence consistently rather than treating it as a separate surprising finding per story.

### Pattern 2: Data Model Divergence

TECH-STORIES.md defines a normalized schema with separate purpose-built tables. The MVP uses a denormalized approach:

| Story Schema | MVP Schema |
|-------------|------------|
| `feedback_submissions` | `feedback` |
| `hotel_scores` (separate table) | `hotels.avg_score`, `hotels.total_feedbacks` (embedded columns) |
| `hotel_rankings` (separate table) | No equivalent; client-side sort by `avg_score` |
| `feedback_triggers` | No equivalent; `bookings.feedback_eligible` column exists |
| `flagged_hotels` (separate table) | No equivalent; implicit via `hotels.status_bucket = 'flagged'` |
| `notifications` (log table) | No equivalent |
| `admin_overrides` (log table) | No equivalent |

### Pattern 3: Rating Scale Divergence

TS-4 and TS-5 specify 1–10 integer sliders. The MVP uses 1–5 star ratings. This affects how scores are interpreted across TS-7, TS-9, and TS-12. All tier threshold values in TS-9 reference a 10-point scale (Top Rated ≥8.0) while the MVP uses a 5-point scale (top_rated ≥4.5). The PRD (SCI-01) says "weighted feedback score" without specifying scale — so both satisfy the PRD.

### Pattern 4: Feedback Dimension Name Divergence

| TS-4 / TS-5 / TS-7 Dimension | MVP Field Name |
|-------------------------------|----------------|
| Cleanliness | `room_cleanliness` |
| Location | — (not present in MVP) |
| Service | `service_quality` |
| Amenities | `amenities_provided` |
| Value | `value_for_money` |
| — | `recommend_to_colleagues` (replaces Location) |

The PRD (COL-01) says "Value, Service, Cleanliness, Amenities, Intent." The TECH-STORIES replace "Intent" with "Location" as the 5th dimension. The MVP uses "Recommend to colleagues" (which matches PRD "Intent"). This is a story-vs-PRD gap on the 5th dimension: TS-4/TS-5/TS-7 say Location; PRD and MVP say Intent/Recommend.

### Pattern 5: Weight Validation Rule Divergence

TS-12 specifies each weight is in range 0.0–5.0 independently, with a rule that "all weights zero" is rejected. The MVP validation requires weights to **sum to exactly 1.0**. This is a minor gap — both prevent degenerate input, but the constraint is different.

## Dependency Graph Validation

The Story Summary table at the end of TECH-STORIES.md lists these dependencies:

```
TS-1: —
TS-2: TS-1, TS-12
TS-3: TS-2, TS-4
TS-4: TS-5, TS-2
TS-5: TS-4, TS-6, TS-7
TS-6: —
TS-7: TS-5, TS-6, TS-12
TS-8: TS-7, TS-6
TS-9: TS-8, TS-11, TS-6
TS-10: TS-6, TS-8, TS-11
TS-11: TS-6, TS-9, TS-10
TS-12: TS-6, TS-2, TS-7
TS-13: TS-5, TS-6, TS-10
```

**Circularity check:**
- TS-4 depends on TS-5 AND TS-5 depends on TS-4: this is a mutual dependency. In dependency terms, the form calls the API, and the API is called by the form — they are tightly coupled. This is not a circular dependency in the implementation sense (it is a producer-consumer relationship), but the dependency listing is slightly misleading. TS-4 should list TS-5 as a "calls" dependency (TS-4 calls TS-5's endpoint). TS-5 lists TS-4 as "called by" (not a build dependency). The Story Summary table should list TS-5's dependency on TS-4 only in the TS-4 row (TS-4 calls TS-5), not in the TS-5 row (TS-5 does not depend on TS-4 to be built). **Minor gap — misleading dependency direction.**

- TS-11 lists TS-9 and TS-10 as dependencies. TS-9 lists TS-11 as a dependency. TS-10 lists TS-11 as a dependency. This means: TS-11 cannot be built until TS-9 and TS-10 are built, but TS-9 and TS-10 cannot be built until TS-11 is built. This IS a genuine circular dependency. **Minor gap — circular dependency in TS-9, TS-10, TS-11.** The resolution: TS-11 depends on TS-6 for storage only; TS-9 and TS-10 read from TS-11's output but TS-11 does not depend on TS-9 or TS-10 to implement the flag API. TS-11's §6 Dependencies listing TS-9 and TS-10 should be removed — they are consumers of TS-11, not prerequisites.

- TS-12 lists TS-2 as a dependency. TS-2 lists TS-12 as a dependency. This is also a circular dependency — TS-2 needs TS-12 config at runtime, and TS-12 references TS-2 as a consumer. TS-12 should not list TS-2 as a build dependency; the relationship is that TS-2 reads from TS-12. **Minor gap — misleading reverse dependency in TS-12.**

## Common Pitfalls for Executor

### Pitfall 1: Conflating "Planned, not built" with "Critical gap"
Many TS features (task queues, RBAC, signed tokens, real notification dispatch) are NOT built in MVP but they align with v3 deferred requirements (AUTH-01, NOT-01). These must be classified as "Planned, not built" — not Critical gaps.

### Pitfall 2: Applying the wrong threshold scale
The story says "< 2.0 threshold" for flagging. The MVP uses < 2.0 out of a 5-point scale. The PRD (SCI-04) says "< 2.0" without specifying scale. These are numerically the same threshold value but the underlying scale differs. The report should note this rather than declaring one wrong.

### Pitfall 3: Treating the "Location" dimension as a Critical gap
TS-4/TS-5/TS-7 use Location as a dimension. The PRD (COL-01) says "Intent." The MVP uses "Recommend to colleagues" which maps to Intent. This is a story-vs-PRD misalignment (the stories drifted from the PRD), but it is Minor because the MVP and PRD agree.

### Pitfall 4: Over-expanding scope to v2 design requirements
DSG-01 through DSG-06 are Phase 7/8 design implementation requirements. They do not appear in TECH-STORIES.md at all. This is expected and correct — TECH-STORIES.md was scoped to the functional feedback intelligence loop, not UI design tokens or Tailwind configuration. Do not flag design requirements as Critical gaps.
