<objective>
Fix the three database-layer issues that cause the Settings page to crash: add a DB uniqueness constraint to `feedback_config` preventing duplicate rows, switch the settings read from `.single()` to `.maybeSingle()` so it never throws PGRST116, and switch the settings save from `.update().eq("id", ...)` to `.upsert({ onConflict: 'singleton' })` so saves are idempotent.

Purpose: The Settings page currently crashes with "JSON object requested, multiple (or no) rows returned" (PGRST116) on any environment that was seeded more than once or set up fresh. This is a three-part fix: DB constraint (prevents new duplicates), migration SQL (removes existing duplicates), and application code (handles edge cases gracefully).

Output: Settings page loads reliably on fresh and multi-seeded databases; saving settings is always safe.
</objective>

    Create the file `basic/10_singleton_constraint.sql` with the following idempotent migration SQL. This file must be version-controlled alongside the existing schema files in `basic/`.

    The migration does three things in order:

    1. **Delete duplicate rows** — keep only the newest row. Safe to run even if only one row exists (the `NOT IN` subquery will simply match the only row, so nothing is deleted).

    2. **Add sentinel column** — `singleton BOOLEAN NOT NULL DEFAULT TRUE`. The `IF NOT EXISTS` guard makes this idempotent.

    3. **Add uniqueness index** — `CREATE UNIQUE INDEX IF NOT EXISTS feedback_config_singleton_idx ON feedback_config (singleton)`. This enforces max one row where `singleton = TRUE` at the DB level. The `IF NOT EXISTS` guard makes it safe to re-run.

    ```sql
    -- Migration: feedback_config singleton constraint
    -- Purpose: Enforce exactly one row in feedback_config
    -- Safe to run multiple times (all statements are idempotent)

    -- Step 1: Remove duplicate rows, keep newest
    DELETE FROM feedback_config
    WHERE id NOT IN (
      SELECT id FROM feedback_config ORDER BY created_at DESC LIMIT 1
    );

    -- Step 2: Add boolean sentinel column
    ALTER TABLE feedback_config
      ADD COLUMN IF NOT EXISTS singleton BOOLEAN NOT NULL DEFAULT TRUE;

    -- Step 3: Enforce uniqueness at DB level
    CREATE UNIQUE INDEX IF NOT EXISTS feedback_config_singleton_idx
      ON feedback_config (singleton);
    ```
